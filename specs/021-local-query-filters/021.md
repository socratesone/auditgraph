# 021 — Local Query Filters & Aggregation

## Project Context

### What is Auditgraph?

Auditgraph is a **local-first, deterministic personal knowledge graph** (PKG) CLI tool for engineers. It ingests plain-text notes, code, PDFs, and DOCX files; deterministically extracts entities and claims; builds explainable links between them; and provides CLI-first navigation — all with stable IDs, audit trails, and reproducible outputs.

The project is at **v0.1.0**, early-stage, not yet on PyPI. It is installed from source and run via the `auditgraph` CLI. It also exposes 22 MCP (Model Context Protocol) tools for LLM-assisted workflows.

**Stack**: Python 3.10+, PyYAML, neo4j driver, pypdf, python-docx, spaCy, dulwich (pure-Python git).

### Core Pipeline

The pipeline runs in stages, each producing deterministic artifacts on disk:

```
ingest → extract → normalize → link → index → query
```

All artifacts live under `.pkg/profiles/<profile>/` in a sharded JSON file structure:

```
.pkg/profiles/<profile>/
  entities/<2-char-shard>/ent_<sha256>.json     # one file per entity
  links/<2-char-shard>/lnk_<sha256>.json        # one file per link
  indexes/bm25/index.json                        # inverted index (single file)
  indexes/graph/adjacency.json                   # adjacency map (single file)
  indexes/git-provenance/file-commits.json       # file→commit mapping
  chunks/<shard>/chk_<sha256>.json
  documents/<doc_id>.json
```

## The Problem

The local query layer has **zero reusable filter, sort, or aggregation primitives**. Every query operation is a purpose-built function with no parameterization:

| What you want | What exists today |
|---------------|-------------------|
| "Show all commits" | Scan 123K entity files, check `type` field on each |
| "Commits by author X" | No operation — not possible without code |
| "Most recent 10 commits" | No sort, no limit — full scan only |
| "How many entities per type?" | No aggregation — count manually |
| "Neighbors of type `authored_by` only" | `neighbors()` returns all edge types; filter manually |
| "Entities where confidence > 0.8" | No field predicates |

This is the **fundamental gap** blocking natural-language query translation. An LLM skill that converts English to graph queries has nothing to target — the MCP tools accept no filter parameters, so every question requires full scans and post-processing that the LLM cannot express.

### Performance Impact

Without indexes, a type-filtered query requires loading all 123K entity files. There is no `load_entities_by_type()`. There is no `load_links()` at all — the only link iteration in the codebase is a raw `rglob("lnk_*.json")` that walks 1.5M files.

---

## Strategy

### Approach

Add **parameterized filters** to the existing query and navigation operations, backed by **lightweight indexes** that avoid full scans. This is not a query language — it is structured parameters on existing CLI commands and MCP tools.

### Implementation Layers

```
Layer 1: Indexes (type index, link-type index)
    ↓
Layer 2: Storage functions (load_entities_by_type, load_links, load_links_by_type)
    ↓
Layer 3: Filter engine (predicate parsing, matching, sorting, pagination)
    ↓
Layer 4: CLI integration (--type, --where, --sort, --limit on query/list/neighbors)
    ↓
Layer 5: MCP integration (new parameters on ag_query, ag_neighbors; new ag_list tool)
```

Each layer depends only on the one below it. Layers 1–2 are infrastructure with no user-visible behavior. Layers 3–5 expose the capability.

### New CLI Command

`auditgraph list` — browse entities without requiring a keyword query. Supports `--type`, `--where`, `--sort`, `--limit`, `--offset`, `--count`, `--group-by`. This fills the gap where `query` requires a search term.

### New MCP Tool

`ag_list` — same as the CLI command, for LLM workflows. Accepts `type`, `where`, `sort`, `limit`, `offset`, `count`, `group_by` parameters.

### What Changes on Existing Tools

| Tool | New Parameters |
|------|----------------|
| `auditgraph query` (CLI) | `--type`, `--where`, `--sort`, `--desc`, `--limit`, `--offset`, `--count`, `--group-by` |
| `auditgraph neighbors` (CLI) | `--edge-type`, `--min-confidence` |
| `ag_query` (MCP) | `type`, `where`, `sort`, `limit`, `offset` |
| `ag_neighbors` (MCP) | `edge_type`, `min_confidence` |

All new parameters are optional. Existing behavior is preserved when they are omitted.

## Open Design Decisions

### Decision 1: Where-Clause Syntax

**Question**: Should `--where` use a simple `field=value` syntax, or support nested field access (e.g., `provenance.created_by_rule=extract.code_symbols.v1`)?

**Context**: Entity JSON has a mix of flat and nested fields. Top-level fields like `name`, `type`, `canonical_key`, `author_email` cover most query needs. But some useful fields are nested: `provenance.created_by_rule`, `refs[0].source_path`, `object.decision` (on claim entities).

#### Option A: Flat top-level fields only

Syntax: `--where "field=value"`, `--where "field>=value"`

Supported operators: `=`, `!=`, `>`, `>=`, `<`, `<=`, `~` (substring contains)

```bash
auditgraph list --type commit --where "author_email=alice@example.com"
auditgraph list --type ner:person --where "mention_count>=5"
auditgraph list --where "name~config"
```

**Pros:**
- Simple to implement — `entity.get(field)` with direct comparison
- Simple to parse — regex split on first operator
- Covers 90%+ of practical queries (most useful fields are top-level)
- Easy for LLMs to generate — no syntax ambiguity
- Matches the mental model of "entity has field X with value Y"

**Cons:**
- Cannot filter by `provenance.created_by_rule` or `refs[0].source_path`
- Users who need nested access must use `jq` post-processing
- If entity schemas evolve to nest more fields, this becomes limiting

#### Option B: Dotted path syntax (JSONPath-lite)

Syntax: `--where "provenance.created_by_rule=extract.code_symbols.v1"`

```bash
auditgraph list --where "provenance.pipeline_version=1"
auditgraph list --where "refs.0.source_path~README"
```

**Pros:**
- Full access to any field in the entity JSON
- Future-proof — works regardless of schema nesting depth
- Familiar syntax (similar to jq `.field.subfield`, MongoDB dot notation)

**Cons:**
- More complex parser — must handle dots in values, array indexing, missing intermediate keys
- `provenance.created_by_rule` is not a common query; the complexity serves edge cases
- Harder for LLMs to generate correctly — dot-path ambiguity increases error rate
- Array access (`refs.0.source_path`) is unintuitive; most users expect `refs[0].source_path`

#### Option C: Flat now, dotted later (phased)

v1 ships flat top-level only. The parser is designed to be extensible. A v2 adds dotted paths if demand materializes.

**Pros:**
- Ships faster with simpler code
- Real usage data informs whether dotted paths are actually needed
- No throwaway work — the flat parser is a subset of the dotted parser

**Cons:**
- If dotted paths are needed immediately, v1 is insufficient
- Two rollouts for one feature

**Recommendation: Option C (flat now, dotted later).** The top-level fields cover the practical query space. The 13 entity types have their most useful fields at the top level (`name`, `type`, `canonical_key`, `author_email`, `authored_at`, `sha`, `mention_count`, `email`). Dotted paths are a power-user feature that can be added without breaking changes.

### Decision 2: Index Format

**Question**: Should the type index and link-type index be single monolithic JSON files or sharded into one file per type?

**Context**: The type index maps `type_string → [entity_id_1, entity_id_2, ...]`. With 13 entity types and 123K entities, the index is modest. But the link-type index maps `link_type → [link_id_1, ...]` and with 1.5M links, the index could reach 50–100 MB.

#### Option A: Single monolithic file per index

```
indexes/types/manifest.json       →  {"commit": ["ent_abc", ...], "ner:person": ["ent_def", ...]}
indexes/link-types/manifest.json  →  {"modifies": ["lnk_abc", ...], "CO_OCCURS_WITH": ["lnk_def", ...]}
```

**Pros:**
- Atomic reads/writes — no partial-update race conditions
- Simple implementation — `json.load()` / `json.dump()`
- Consistent with existing index format (BM25 is a single 38 MB file)
- Easy to diff and audit

**Cons:**
- The link-type index at 1.5M entries could be 50–100 MB — expensive to load when you only need one type
- Rebuild requires holding entire index in memory
- Write amplification — changing one type rewrites the entire file

#### Option B: One file per type

```
indexes/types/commit.json         →  ["ent_abc", "ent_def", ...]
indexes/types/ner_person.json     →  ["ent_ghi", ...]
indexes/link-types/modifies.json  →  ["lnk_abc", ...]
```

**Pros:**
- Load only the type you need — `load_entities_by_type("commit")` reads one small file
- Link-type index becomes manageable — `CO_OCCURS_WITH.json` (1M entries, ~30 MB) is loaded only when needed
- Parallel rebuild — each type can be built independently
- Better scaling trajectory as entity/link counts grow

**Cons:**
- More files to manage — listing types requires `glob("indexes/types/*.json")`
- No atomic snapshot of "all types" — must read multiple files
- Type names with special characters (e.g., `ner:person`) need filename sanitization (colon → underscore)
- Slightly more complex rebuild logic

#### Option C: Monolithic entity index, sharded link index

Hybrid: the type index (13 types, 123K IDs) stays monolithic. The link-type index (7 types, 1.5M IDs) shards per type.

**Pros:**
- Right-sizes each index — the small one stays simple, the large one stays efficient
- Entity type index at 123K entries is ~3–5 MB, cheap to load whole
- Link type index only loads the relevant type file

**Cons:**
- Two different patterns — inconsistent, harder to document
- Must decide where the size threshold is for switching strategies

**Recommendation: Option B (one file per type).** The consistency of a single pattern outweighs the minor complexity. Filename sanitization is trivial (`ner:person` → `ner_person.json`). The per-type approach scales naturally and avoids the 50–100 MB monolith problem for links. It also enables a useful operation for free: listing all types is just `ls indexes/types/`.

### Decision 3: Default MCP Limit

**Question**: Should `ag_list` (and `ag_query`) have a default result limit when called via MCP, to prevent accidentally flooding an LLM's context window with 123K entities?

**Context**: MCP tool results go directly into an LLM's context window. A query that returns 42,157 `ner:org` entities as JSON would consume millions of tokens and either crash the client or produce garbage. The CLI has no such constraint — terminal output can be piped, paginated, or redirected.

#### Option A: No default limit (same as CLI)

MCP tools return all matches. The calling LLM is responsible for specifying `limit`.

**Pros:**
- Consistent behavior between CLI and MCP
- No "magic" defaults that surprise users
- LLMs can request exactly what they need

**Cons:**
- A single missing `limit` parameter causes a catastrophic context overflow
- LLMs frequently forget optional parameters, especially on first use
- Debugging is painful — the tool appears to "hang" or crash without explanation
- Violates the principle of safe defaults

#### Option B: Default limit of 100 for MCP, unlimited for CLI

MCP tools default to `limit=100`. CLI defaults to unlimited. Both can be overridden explicitly.

**Pros:**
- Safe by default — worst case is 100 results, which is manageable in any context window
- CLI users get full output for piping/scripting
- LLMs can override with `limit=1000` or `limit=0` (unlimited) when they know what they're doing
- Follows the principle of least surprise for LLM integrations

**Cons:**
- Behavioral difference between CLI and MCP for the same logical operation
- LLMs may not realize they're getting truncated results unless the response includes a `truncated: true` / `total_count: N` field
- The number 100 is arbitrary — may be too low for some use cases, too high for others

#### Option C: Default limit of 100 for MCP, with total count in response

Same as Option B, but every response includes `total_count` (the full count before limiting) and `truncated: true/false`.

```json
{
  "results": [...],
  "total_count": 42157,
  "limit": 100,
  "offset": 0,
  "truncated": true
}
```

**Pros:**
- All benefits of Option B
- LLM knows results are incomplete and can request more if needed
- Enables pagination workflows: "show next 100"
- `total_count` is useful for aggregation even without full results

**Cons:**
- Computing `total_count` requires scanning all matches even when limiting — some performance cost
- Slightly more complex response format

**Recommendation: Option C (default limit of 100 with total count).** The total count is cheap to compute (it's a `len()` call on the filtered list before slicing) and dramatically improves the LLM's ability to navigate large result sets. The `truncated` flag makes the behavior transparent rather than silently lossy.

### Decision 4: Reverse Adjacency Index

**Question**: Should this spec include a reverse adjacency index (`to_id → [edges]`) to enable inbound neighbor queries, or defer it to a separate spec?

**Context**: The current adjacency map is keyed by `from_id` — you can ask "what does node X point to?" but not "what points to node X?". For a file entity, you cannot efficiently find "which commits modified this file?" without scanning all links. This is a common and valuable query pattern:

- "Who authored this file?" (inbound `authored_by` via `modifies`)
- "What tags point to this commit?" (inbound `tags`)
- "What entities co-occur with this person?" (inbound `CO_OCCURS_WITH`)

#### Option A: Include in this spec

Add FR for a reverse adjacency index at `indexes/graph/reverse-adjacency.json`. Add `--direction in|out|both` to `neighbors`.

**Pros:**
- Completes the graph navigation model — without it, half of graph queries are impossible locally
- The implementation is straightforward — same structure as forward adjacency, keyed by `to_id`
- The `neighbors` command already exists; adding `--direction` is a natural extension
- Maximizes the value of the filter infrastructure built in this spec

**Cons:**
- Increases scope — another index to build, maintain, and test
- The adjacency index is currently empty (2 bytes) across all profiles — the forward adjacency must be fixed first before reverse makes sense
- Risk of scope creep delaying the core filter/sort/limit features

#### Option B: Defer to separate spec

This spec focuses on filtering, sorting, and aggregation. Reverse adjacency gets its own spec (022 or similar).

**Pros:**
- Keeps this spec focused on its core mission: parameterized queries
- Allows forward adjacency to be fixed and validated first
- Separate spec can properly scope inbound traversal, path finding, and bidirectional BFS

**Cons:**
- Delays a critical query capability
- The `neighbors --edge-type` filter added here is only half-useful without inbound traversal
- A second spec means a second implementation cycle

#### Option C: Defer the spec, but design for it now

This spec does not implement reverse adjacency, but the adjacency rebuild code and `neighbors` command are designed with explicit hooks for adding it later. Specifically: the adjacency format includes a `direction` field on edges, and the `neighbors` function accepts a `direction` parameter that currently only supports `"out"`.

**Pros:**
- No scope increase — ships the same features
- No throwaway work when reverse adjacency lands
- The `direction` parameter in the API signals to users that inbound is coming

**Cons:**
- A parameter that only accepts one value (`"out"`) is slightly misleading
- Designing for a future feature risks over-engineering if the reverse index design changes

**Recommendation: Option B (defer), with one caveat.** The reverse adjacency index depends on the forward adjacency actually being populated — which it currently is not. Fixing the forward adjacency rebuild is prerequisite work that belongs in this spec (or is already in scope via the link-type index). Once that works, reverse adjacency is a clean follow-on. Do not add a `direction` parameter that only accepts one value — that is speculative API surface. Instead, document in this spec's "Future Work" section that reverse adjacency is the natural next step.

## Summary of Recommendations

| Decision | Recommendation | Rationale |
|----------|---------------|-----------|
| Where-clause syntax | Flat fields only (v1) | Covers 90%+ of queries; dotted paths add complexity for edge cases |
| Index format | One file per type | Consistent pattern, scales naturally, avoids monolithic link index |
| Default MCP limit | 100 with `total_count` | Safe default + transparent truncation enables LLM pagination |
| Reverse adjacency | Defer to separate spec | Forward adjacency must work first; clean separation of concerns |
